# 緣起：為什麼自己做講義網站
> 與其用傳統 PPT，不如用 Vibe Coding 打造一個量身打造、可長期維護的講義工具

## 一個備課時冒出的念頭

### 💡 既然在講 AI 寫程式，講義也用 Vibe Coding 做
- 與其用傳統 PPT，不如用 Vibe Coding 打造專屬講義網站
- 一個晚上完成模板雛形：電腦、平板、手機都能看，還能匯出 PDF
- 封裝成 Agent Skill：之後只要提供初稿，AI 就生成講義網頁

### 🔁 為什麼一直在換新的 AI 工具？
- 刻意培養「多個 AI Agent 交替使用」的能力
- 只要某個 AI 被捧上天，就注定不久後會隕落，歷史總是驚人地相似
- 工具間的差異沒你想的大，重點是用案例理解如何解決痛點

> **不要為了折扣付年費**
> 教學的目標從來不是強調「哪一款工具最強」，而是透過實際案例，理解怎麼解決痛點。
>
> 因為每個時期最好用的 AI 一直在改變，把能力綁在單一工具上，是最危險的事。

# 安裝 Codex：在終端機生成第一個網頁
> 準備工作區 → 確認 Node.js → 安裝 Codex → 一段 Prompt 生成講義網頁

## 環境準備

### 🗂️ 給 AI Agent 一個乾淨的工作區
- 先準備一個空資料夾，讓 AI Agent 有明確的工作範圍
- 編輯器用 VSCode / Cursor / Antigravity 都可以
- 沒偏好的話可用 Google 免費的 [Antigravity](https://antigravity.google)，這次只拿它來檢視程式碼變更

### 🟢 確認 Node.js 版本
- Codex 與專案會用到 Node.js，先確認電腦有安裝
- 沒有版號就先裝；建議用 NVM 管理、切換多個版本

```prompt [label="確認 Node 版本"]
node -v
```

```prompt [label="Mac 安裝 NVM"]
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | bash
```

[flow]
1. 安裝 NVM — Mac 用上方指令；Windows 下載 [nvm-windows](https://github.com/coreybutler/nvm-windows/releases) 的 exe
2. nvm install --lts — 安裝最新 LTS 版的 Node.js
3. node -v — 確認版號，代表安裝成功
[/flow]

## 安裝與登入 Codex

### ⬇️ 全域安裝 Codex

```prompt [label="安裝 Codex"]
npm i -g @openai/codex
```

### 🔑 三種登入方式
- 安裝完在終端機輸入 `codex` 啟動
- 第一次會詢問是否信任資料夾，選 Yes 才能繼續

[tags]
- [green] ChatGPT 帳號登入：最推薦，開網址登入即可
- [blue] 裝置代碼授權：適合無瀏覽器的 VM（需先到設定開啟）
- [orange] OpenAI API Key：依用量計價，沒設上限小心帳單
[/tags]

## 用一段 Prompt 生成講義網頁

### 🎯 Prompt 設計三要素

[flow]
1. 目標 — 只生成單純的 HTML 網頁，方便日後部署
2. 風格 — 說明想要的設計感，沒想法可讓 AI 扮演設計師
3. 架構 — 分階段描述：開頭、層級設計、社群連結、側邊欄目錄
[/flow]

```prompt [label="生成講義網頁"]
請扮演熟悉 AI 主題授課的講師，以「Agent Skills」為主題設計講義
講義用純 html 網頁（CSS/JS 都放裡面）呈現
日系風格的設計感，部分文字與背景可以搭配適當的漸層
開頭會根據內容呈現標題，然後下面有發人深省的標語
會根據不同層級與意圖呈現對應的區塊（ex: 大小標題、陳述、列點、流程）
最後放上社群連結鼓勵大家訂閱（FB/YouTube/Medium）
側邊欄會有對應的目錄，方便跳轉到對應位置
```

# 痛點：好看的第一版，卻改不動
> 生成式 AI 的隨機輸出，讓細節調整變成一場惡夢

## 兩個都不太好的選擇

### 😩 對話調整 vs. 直接改 HTML
- AI 生成的描述、段落結構，未必符合你的期待
- 選項一：跟 AI 對話調整 → 隨機性太高，還可能動到不該動的
- 選項二：直接改 HTML → 就算本身是工程師，改起來都覺得痛苦

> **改不動的漂亮，等於沒用**
> 假使後續要花很大力氣才能調整，那初始版本產生得再快、畫面再漂亮，也很難用到現實情境。
>
> 既然生成式 AI 的輸出注定充滿隨機性，那就換個思路：有沒有方法，能在結果產生後輕鬆調整細節？我的答案是「建立模板」。

# 建立可重複使用的模板
> 不要每次從零生成，而是建立模板；之後只要放進不同內容，就能產出穩定一致的成果

## 把 HTML 逆向成 Markdown

### 🧱 模板的核心概念
- 不要讓 AI 每次從零生成，而是建立一套「模板」
- 之後只要替換內容，就能產出穩定、一致、可控的成果
- 不只簡報、寫文件、建網頁，任何重複性工作都適用

### 🔄 一段 Prompt，把網頁變成可編輯模板

[flow]
1. 期待目標 — 讓網頁可以透過 Markdown 編輯內容
2. 設計架構 — 拆解結構（列點/步驟/圖片/表格）、分層次（大中小標），以常見 Markdown 語法設計規則
3. 渲染方式 — 建立腳本，編輯 Markdown，HTML 即時更新
4. 說明文件 — 操作寫進 README.md，並建立 package.json
[/flow]

```prompt [label="逆向成 Markdown 模板"]
我希望剛剛生成出來的網頁，是可以透過 Markdown 格式編輯內容的
請幫我將網頁結構拆解（列點、步驟、圖片、表格、註解）、分層次（大標、中標、小標），並以常見的 Markdown 語法來設計規則
最後建立一個腳本，我只要編輯 Markdown 文件，網頁的 HTML 就會及時改變
操作方式寫在 README.md 文件中，並建立 package.json
```

> **想要掌控力，就要把內容放在可維護的格式裡**
> 把內容設計在「可編輯、好維護」的 Markdown，而不是放在最終輸出的 HTML 裡。
>
> 接著用 `npm run dev` 啟動，編輯 Markdown，就能看到網頁即時更新。

## 封裝成 Agent Skill

### 🧩 為什麼要封裝成 Skill
- 把剛剛驗證過的工作流，設計成一個可重複使用的 Agent Skill
- 第一次接觸 Agent Skills？可參考[這支教學影片](https://youtu.be/xe00zJEtuMo)
- SKILL.md 的 `name` 是技能名稱，`description` 說明觸發時機與預計成果

```prompt [label="建立 Agent Skill"]
根據下面需求在「.agents」資料夾建立 Skill：
- 參考模板：根據剛剛腳本設計的規格，建立明確的 Markdown 使用規範
- 觸發時機：使用者提供初步想法想建立網頁時，會觸發這個 Skill
- 執行順序：這個 Skill 會先深入研究輸入的主題，再根據規範設計對應的 Markdown 文件；最後把 Markdown 文件透過腳本生成 html
- 特別提醒：預計會有多個主題，所以觸發時，請先建立新的資料夾，裡面存放新生成的 Markdown 與 HTML 檔案；並且建立獨立的 scripts 與 references 資料夾，確保可單獨執行
```

### ✅ 驗證 Skill：一句話生成講義
- 在對話視窗丟出主題，AI 就會自動觸發對應的 Skill

```prompt [label="驗證 Skill"]
我想要建立深度介紹「ChatGPT 使用技巧」的課程講義網頁。
```

> **特殊語法不用背，因為它寫在 Skill 裡**
> 模板可以設計專屬的特殊語法，但你不需要去記。
>
> 你只要提出想放進講義的資訊，AI 就會辨識內容、找到最合適的版型整合進去；你要做的，只是在文字不符預期時做微調。

# 自由的代價：功能的不完善
> 自己設計模板，代表你是這套工具目前唯一的維護者

## 實際用過，才知道哪裡要補

### ⚠️ 看似厲害的模板，其實一堆缺口
- 側邊欄方便導覽，但佔掉展示空間，希望講課時能自由收合
- 預設淺色設計，但關燈授課時，螢幕白光會讓台下眼睛不舒服
- 有些單位要 PDF 講義，但網頁預設列印出來的畫面非常醜

> **你想要主控權，就得承擔客製化的複雜度**
> 模板的細節，是在實際使用後才知道哪裡要調整。設計與程式可以交給 AI，但方向與美感得自己掌握。
>
> 如果只是要線上簡報工具，市面上已有很多現成方案。自建模板，與其說是更好的方案，不如說是設計一套最符合自己需求的工具。

## 從專業角度補完細節

### 🔍 OG：公開分享前一定要設定
- 若成品要公開，沒設好標題、描述、縮圖，網址預覽畫面會很醜
- 用 [opengraph.xyz](https://www.opengraph.xyz/) 貼上網址，檢查 OG 是否設定完成

### ⚙️ 共用資訊抽成 config
- 講者資訊、頁尾社群宣傳、預設 OG → 抽到 `config/global.yaml` 全頁共用
- OG 圖片放在 `assets/` 資料夾
- 需要客製化的資訊，用各課程的 `config.yaml` 蓋過去

> **用系統設計的角度思考模板**
> 變動性高的資訊放在 Markdown 方便修改；共用的資訊設計成 config 維護；Markdown、網頁、OG 圖片的初始化，則交給 Agent Skill 處理。
>
> 這樣 AI 能幫你快速完成第一版，而你也能在後續輕鬆調整細節。

# 輕鬆部署上線
> 因為產出是單純的 HTML，可以直接用 GitHub Pages 託管

### 🚀 從 Init Repository 到網址上線

[flow]
1. Init Repository — 在編輯器初始化 Git，公開可選 Public
2. Generate & Push — 自動生成 commit，推送到 GitHub
3. Settings → Pages — 用 Deploy from a branch，Branch 切到 main
4. 加上路徑 — 網頁在 example 資料夾，網址後面補上對應路徑即可瀏覽
[/flow]

> **沒有技術背景，也可能踩到坑**
> 雖然整個專案是 Vibe Coding 出來的，但完全沒有技術背景，可能會設計出不合理的架構、忽略 OG 這類細節。
>
> 當你開始思考專案架構，Vibe Coding 的成果才會慢慢脫離 Demo，進入長期可用的狀態。

# 結語：知道自己要什麼，然後動手做
> 回顧這趟流程，我們其實只做了三件事

[summary]
- 🛠️ **Codex 從零生成** | 用一段結構清楚的 Prompt，讓 Codex 在終端機生成講義網頁
- 🔄 **轉成可維護模板** | 把 AI 的隨機輸出，逆向成可用 Markdown 編輯、腳本即時渲染的模板
- 🧩 **封裝成 Agent Skill** | 把整套工作流打包成 Skill，一句話就能重現，讓它可重複使用
[/summary]

> **從玩具走向產品的關鍵，是「設計」**
> AI 真正厲害的地方，不是幫你省下一次工作的時間，而是讓你有機會把原有的工作流，設計成一套可以持續優化的模板。
>
> 從這一刻起，你不再只是使用別人的工具，而是開始有能力，把工具設計成更符合自己需求的樣子。
