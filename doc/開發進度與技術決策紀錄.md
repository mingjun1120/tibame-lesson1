## 一、技術決策紀錄（ADR）

### ADR-001：採用 React Native，不做原生雙平台開發

- 日期：2026-01-15
- 決策：App 以 React Native 開發單一程式碼庫。
- 理由：前端／App 僅 1 位工程師，原生 iOS + Android 雙開發人力不足；React Native 可共用約 90% 程式碼。
- 取捨：少數需要原生效能的功能（如相機）以原生模組補強。

### ADR-002：後端採用 FastAPI

- 日期：2026-01-16
- 決策：後端 API 框架採用 Python FastAPI。
- 理由：開發速度快、原生支援 async、自動產生 OpenAPI 文件，利於前後端協作。

### ADR-003：金流改採 ECPay 押金代管，放棄自建錢包

- 日期：2026-02-20
- 決策：押金與交易金流串接綠界 ECPay，不自建平台儲值錢包。
- 理由：自建儲值錢包涉及《電子支付機構管理條例》，法遵成本過高且需第三方支付執照；ECPay 押金代管即可滿足 MVP 需求。

### ADR-004：信用評分與平台保險延至 Phase 2

- 日期：2026-04-16
- 決策：F-10 信用評分機制、F-11 平台保險自動理賠移出 MVP，列為上線後 Phase 2。
- 理由：
  1. 保險方案需與保險公司簽約並通過法務審查，時程不可控。
  2. 信用評分演算法需累積實際交易資料後才有意義。
  3. MVP 階段先以「人工審核 + 簡易押金代管」滿足基本信任需求。
- 影響：MVP 開發範圍縮小，有助於趕上順延後的時程。此決策已在 2026-04-15 第二次進度會議正式通過。

## 二、開發週會技術摘要（節錄）

### 2026-03-W2（3 月第 2 週）

- 完成後端專案骨架與資料庫 Schema 第一版。
- 完成手機 OTP 註冊／登入 API（F-01）。
- 前端完成登入與社區綁定畫面。

### 2026-03-W4（3 月第 4 週）

- 完成工具刊登 API 與 S3 圖片上傳（F-03）。
- 完成地圖搜尋附近工具（F-04），整合 Google Maps 與 PostGIS。
- 風險預警：ECPay 押金代管測試環境申請流程比預期久。

### 2026-04-W2（4 月第 2 週）

- ECPay 押金代管串接在測試環境遇到回呼（callback）驗證問題，處理中。
- 平台保險方案的法務審查尚未完成，保險公司合約仍在洽談。
- 初步評估：MVP 開發進度落後約 2～3 週。

### 2026-04-W3（4 月第 3 週）

- 經團隊與 PM 討論，確認將 F-10、F-11 移至 Phase 2（見 ADR-004）。
- ECPay 回呼問題已解決，押金代管基本流程可運作。
- 後續聚焦完成 F-06 借用流程、F-07 訂單聊天、F-09 歸還與評價。

## 三、目前已知技術風險

| 風險 | 影響 | 因應 |
|------|------|------|
| ECPay 正式環境金流串接仍需驗證 | 可能影響上線時程 | 預留 staging 環境完整測試 |
| 工具照片量成長使 S3 成本上升 | 雲端費用增加 | 上線後評估圖片壓縮與生命週期規則 |
| React Native 版本升級相容性 | 維護成本 | 鎖定 0.74，暫不跟進新版 |
